# Product Requirements Document (PRD): 수원러닝크루(SRC) 러닝 기록 및 생존 관리 시스템

## 1. Project Overview
- **프로젝트 명:** 수원러닝크루(SRC) 기록 인증 및 관리 시스템
- **목적:** 멤버들의 러닝 기록 인증, 마라톤 PB 관리, 월간 생존(활동) 여부 자동 계산 및 대시보드 제공
- **플랫폼:** 모바일 최적화 웹 애플리케이션

## 2. Technology Stack
- **Framework:** Next.js (App Router 기반)
- **Backend / DB / Auth:** Supabase (PostgreSQL)
  - **인증:** Supabase Auth (카카오 소셜 로그인 단일 수단)
  - **보안:** Row Level Security (RLS)를 통한 권한 제어
- **Styling & UI:** Tailwind CSS, shadcn/ui (모바일 반응형 최적화)
- **Deployment:** Vercel

---

## 3. User & Authentication (Role System)

### 3.1. 인증 정책 (Authentication)
- **단일 인증 수단:** 이메일/비밀번호 등 타 가입 수단은 철저히 배제하며, **'카카오톡 계정 연동 간편 로그인'**만 가입 및 로그인 수단으로 허용합니다.

### 3.2. 권한 레벨 (Role)
- **WAITING (대기):** 최초 카카오 로그인(가입) 직후의 기본 상태. 관리자의 승인 전까지 어떠한 앱 기능도 사용 불가.
- **REGULAR (정회원):** 운영자 승인 완료 상태. 
  - 권한: 본인 기록 CRUD, 타인 기록 및 누적 거리 조회 가능. (타인의 연락처 등 민감 정보는 마스킹 처리 또는 노출 불가)
- **PACER (페이서):** 정회원 권한 + 회원 목록 조회 시 닉네임 옆에 페이서 전용 이모티콘(예: 🎈) 직관적 노출. (운영자가 임명)
- **ADMIN (운영자):** 시스템상 최소 1명 필수 유지. 
  - 권한: 가입 대기자(WAITING) 승인, 회원 강퇴/면제 처리, 모든 데이터(장소, 타인 기록 등) 수정 권한, 모든 민감 정보 열람 권한.

### 3.3. 데이터 관리 정책 (Soft Delete)
- **사용자 데이터 보존:** 강퇴 또는 탈퇴 시 DB에서 완전히 삭제(Hard Delete)하지 않고 `is_active=false` 형태로 상태만 변경합니다. 추후 카카오 계정으로 재가입하여 승인받을 경우, 기존 누적 거리 기록과 마라톤 PB가 그대로 복구되어야 합니다.

## 4. Core Features

### 4.1. 러닝 인증 시스템
#### 4.1.1. 인증 폼 (Form) 입력 항목
- **거리 (Number):** 소수점 한 자리까지 입력 가능. **최소값 3.0 이상 Validation 필수**.
- **장소 (Dropdown):** ADMIN이 관리하는 장소 목록에서 필수 선택.
  - *장소 Soft Delete 정책:* 운영자가 장소를 삭제하더라도, 기존에 해당 장소로 등록된 인증 기록의 텍스트가 깨지거나 null이 되면 안 됩니다. 장소를 Soft Delete 처리하거나, 인증 시점에 장소명 텍스트를 기록 테이블에 스냅샷으로 별도 저장해야 합니다.
- **날짜 (Date picker):** 기본값은 '오늘'. 미래 날짜는 선택 불가.
- **인증 종류 (Enum):** `PERSONAL` (개인런) / `REGULAR` (벙) - 필수 선택 항목.
- **페이싱 유무 (Boolean):** 본인이 당일 페이서 역할을 했는지 여부 체크 (선택).

#### 4.1.2. 날짜 제한 (Date Restriction)
- **REGULAR / PACER:** 오늘을 기준으로 **최대 30일 전**의 기록까지만 소급하여 입력 가능.
- **ADMIN:** 과거 날짜 제한 없이 무기한 소급 입력 가능.

#### 4.1.3. 기록 관리 권한
- 본인이 작성한 인증 기록은 본인이 직접 수정 및 삭제가 가능합니다. (ADMIN은 모든 기록에 대한 수정/삭제 권한 보유)

### 4.2. 마라톤 PB 관리
- **종목 (고정 카테고리):** 10K, Half, Full
- **수정 권한:** 본인 및 ADMIN

## 5. Core Logic: 월간 생존 (Survival) 대시보드

### 5.1. 대시보드 UI
- 앱 접속 시 즉각적으로 '이번 달 생존 달성 여부', '남은 필요 횟수', '현재 상태'를 직관적으로 표시합니다.

### 5.2. 생존 조건 계산 알고리즘
- **업데이트 기준:** 매월 1일 00:00을 기준으로 과거 데이터 통계를 리셋하고 새로운 월의 뷰로 갱신합니다.
- **[Rule 1] 1일 1회 카운트:** 하루에 2회 이상 인증하더라도 1회 인증한 것으로 병합하여 계산합니다.
- **[Rule 2] 벙(REGULAR) 우선순위:** 동일한 날에 PERSONAL(개인런)과 REGULAR(벙) 인증이 모두 존재한다면, 해당 일자는 REGULAR 참여 1회로 계산합니다.
- **[Rule 3] 최종 생존 조건:** 아래 '조건 A' 또는 '조건 B'를 만족하면 이번 달 **'생존'**으로 처리됩니다.
  - **조건 A:** 월 총 인증 일수 >= 2 **AND** 이 중 REGULAR 인증 일수 >= 1
  - **조건 B:** (벙 참석 없이) 월 총 PERSONAL 인증 일수 >= 6

### 5.3. 상태 면제 (Exemption) 처리
- 부상, 출장 등의 합당한 사유 발생 시, ADMIN이 해당 회원의 상태를 **'면제'**로 수동 변경할 수 있습니다.
- 면제 상태의 회원은 해당 월의 인증 의무에서 제외됩니다.
- 면제 상태는 월이 넘어가도 자동으로 풀리지 않으며, ADMIN이 수동으로 '정상' 상태로 복구할 때까지 계속 유지됩니다.

## 6. AI Vibe Coding Action Items
개발 시 다음 항목들을 우선적으로 처리합니다.

1. **Authentication:** 이메일/비밀번호 로그인을 철저히 배제하고, Supabase Auth를 이용한 **오직 카카오 소셜 로그인만 허용**하는 초기 세팅을 진행합니다.
2. **Database & Security:** Supabase RLS(Row Level Security) 규칙을 엄격히 포함한 데이터베이스 스키마(SQL 또는 ORM)를 작성합니다.
3. **Project Structure:** Next.js App Router 기반의 적절한 폴더 구조를 생성합니다.
4. **UI/UX:** shadcn/ui를 적극 활용하여 모바일 화면에 최적화된 인증 폼 및 대시보드 UI 컴포넌트를 개발합니다.
5. **Business Logic:** 월별 생존 조건(Rule 1~3)을 오차 없이 계산하는 핵심 비즈니스 로직(Utility Function)을 작성하고 발생 가능한 엣지 케이스에 대한 테스트를 수행합니다.
